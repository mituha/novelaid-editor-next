# 猫モフ Apps - 小説執筆アプリを創ろう - 10. ドキュメントエリア

![](./images/20250622_note_1280_670.png)

猫モフ Apps は、猫をモフモフしながら思いついたアイデアを、バイブコーディングでゆるっと創っていく企画です。  

前回はフロントエンドとバックエンドについてで、プログラムの裏側寄りの話題でした。  
今回はもう少し表側、ユーザーが触れる部分の話題、と見せかけてまた構造的な話になります。

## ドキュメントエリア

![](./images/10/10_ドキュメントエリア.png)

オリジナル版の`novelaid-editor`のメイン画面は上の画像のようになっています。  
真ん中のエリアがドキュメントエリアです。  
ドキュメントエリアは左右に分割表示でき、それぞれタブで複数のドキュメントを開くことができます。

画像では左が執筆用のエディター画面で、右側にはプレビュー表示として縦書きでの表示を行っています。  

今回、このドキュメントエリアの機能の実装を行っていきます。
実装するのは、以下の２つの機能です。

1. 複数ドキュメントのタブ表示
2. ドキュメントの左右分割表示

なお、今回はオリジナル版の実装を参考に方向性を決めて実装を行っていきます。

## ドキュメントビューモード

前述のオリジナル版では、右側には小説のプレビュー画面を表示しています。  
このように、将来的にはエディター画面だけでなく、プレビュー画面の実装を行います。
そのための下準備も兼ねて、DocumentViewModeを定義します。  

```typescript
/**
 * ドキュメントエリアで表示するViewの種類。
 */
export type DocumentViewMode = 'none' | 'editor' | 'canvas' | 'reader' | 'preview';

このDocumentViewModeを定義してください。
```

この定義は今後の拡張のためのものです。
純粋に小説執筆機能のみで良いなら不要とも言えます。もっとも、そのような場合は既に自分用の小説執筆アプリは完成していると言っても良いでしょう。
このDocumentViewModeをどのように使うかですが、以下のようになります。
これをアプリケーション仕様に追加しておいてください。

```markdown

#### ドキュメントビュータイプ

ドキュメントを表示するビューの種類を表します。　　
ドキュメントタイプにより利用できるビューが若干異なります。

##### ビューの種類

* Editor (`editor`)
  + ドキュメント(テキスト)を編集するビュー
  + Monaco Editorを使用
  + 基本的にはこれがドキュメントのメインのビューとなります
    + ドキュメントを選択した際、通常はこのモードで開きます
  + 左右どちらのドキュメント領域でも利用可能です
* Canvas (`canvas`)
  + ドキュメントをGUIで編集するビュー
  + Monaco Editorを利用しない専用の編集機能を持つビュー
  + `chat`であれば、チャット用ビューが利用可能です
  + 通常、Canvas時はReaderはサポートしません
* Reader (`reader`)
  + ドキュメントを閲覧するビュー
  + ドキュメントタイプにより表示が若干異なります
    + 例えば、`novel`であれば縦書き、横書きの閲覧用ビューが利用可能です
    + `markdown`であれば、マークダウンのレンダリングビューが利用可能です
    + `image`であれば、画像の表示ビューが利用可能です
  + 通常、EditorとReaderは同じ領域での切り替え式となります
  + ユーザーインターフェース上は「閲覧」として扱われ、Canvasと排他的な機能として提供されます
    + 同一領域内で「編集」と「閲覧」の切り替えボタンによりモードを遷移します
* Preview (`preview`)
  + ドキュメントをプレビューする別タブのビュー
  + 表示内容としては、ReaderまたはCanvasと同じです。
  + Readerが同一領域で切り替え式であるのに対し、Previewは常にEditorと並べて表示されます。
  + また、Editorが閉じられた際には連動して閉じられます。
  + Editorを持たないドキュメントではサポートされません。

##### ドキュメントタイプの対応

| ドキュメントタイプ | Editor | Canvas | Reader | Preview |
| --- | --- | --- | --- | --- |
| novel     | ◎ | x | ○ | ○ |
| markdown  | ◎ | x | ○ | ○ |
| chat      | ○ | ◎ | x | ○ |
| image     | x | x | ◎ | x |
| git-diff  | ◎ | x | x | x |
| browser   | x | ◎ | x | x |
| unknown   | ◎ | x | x | x |

◎: 規定のビュー
○: 利用可能
x: 利用不可

```

オリジナル版の`novelaid-editor`では、上記のようにドキュメントエリアのビューのモードを切り替えられるようになっています。  
なお、これは概念的なモードで、実際のViewのコンポーネントは`DocumentType`と`DocumentViewMode`の組み合わせで決まります。
`novel`の`preview`は小説閲覧用の縦書き表示ビューであり、`markdown`の`preview`はマークダウンのレンダリングビューとなります。

## 複数タブの表示

次に、複数タブでの表示に対応します。

```markdown

#### ドキュメントビューエリア

VSCodeのドキュメントビューエリアと同様の機能を提供します。  
ドキュメントビューは、複数のドキュメントをタブで開けるようにします。
将来的にドキュメントエリアを左右に分割して２つのドキュメントを並べて表示できるようにします。
複数のドキュメントの保持はDocumentContextで管理します。
DocumentContextのopenDocumentで、指定ファイルのドキュメントを開けるようにします。

```

こんな感じでタブ対応を実装してもらいます。が、もう少し具体的な指示も追加しておきます。

```markdown

また、ドキュメントの管理は以下のDocumentStateで行って下さい。

export interface DocumentState {
    path: string;
    content: string;
    metadata: Record<string, any>;
    documentType: NovelaidDocumentType;
    isDirty: boolean;
    // 各ペイン・各スロット（メイン/プレビュー）の表示状態。'none'ならタブが表示されない。
    leftMainView: DocumentViewMode;
    rightMainView: DocumentViewMode;
    leftPreviewView: DocumentViewMode;
    rightPreviewView: DocumentViewMode;
}

export interface TabItem {
    path: string;
    isPreview: boolean;
}

```

これだけの情報でどのように実装してくれるかは面倒なのでAIガチャと洒落込みます。

AIが下記のような実行計画を立ててくれるのですが、案の定、思ったような実装予定にはなりませんでした。

```
[NEW] 
DocumentContext.tsx
ドキュメントの状態管理を行う React Context を作成します。

State:
documents: DocumentState[]: 開いている全ドキュメントの実体。
leftTabs: TabItem[]: 左ペインのタブリスト。
rightTabs: TabItem[]: 右ペインのタブリスト。
activeLeftPath: string | null: 左ペインでアクティブなドキュメントのパス。
activeRightPath: string | null: 右ペインでアクティブなドキュメントのパス。
```

ここで何が想定と違ったかというと、leftTabs, rightTabsの定義です。
ある意味、オーソドックスで妥当な実装ですが、今回のDocumentStateの定義の意図からは外れます。  
とりあえず、何を意図してDocumentStateを定義したかについて解説します。

通常の実装ですと、ドキュメントの配列(documents)と左右のペインで開いているタブの配列とを別々に管理します。
つまりはAIの実行計画通りの実装です。
しかし、今回のDocumentStateでは、ドキュメントが開いているペインの情報を含んでおり、タブの配列を管理する必要がないようにする想定でした。  

これは、DocumentStateがどのタブにも表示されなくなった場合に破棄しやすくするためです。
なお、この方式によりDocumentStateのライフサイクル管理を簡単にするのが目的です。
また、タブの順序や並び替えはサポートしないものと割り切っています。

というようなことをAIに追加で説明して、それなりに理解したような実行計画が出たところで実装しました。
